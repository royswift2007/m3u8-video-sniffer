# M3U8D 安装包方案实施级定稿清单

## 1. 文档定位

本文档为 [`M3U8D安装包方案实施级定稿清单.md`](plans/M3U8D安装包方案实施级定稿清单.md) 的实施级定版稿，用于统一后续安装包改造范围、职责边界、落地顺序与验收口径。

本文档仅覆盖安装包方案与相关实现边界，不涉及业务功能扩展。

## 2. 已确定方案总则

### 2.1 打包与安装器组合

- 桌面程序采用 PyInstaller + Inno Setup 方案。
- PyInstaller 采用 one-dir 产物形态，不采用 one-file。
- Inno Setup 负责安装器封装、安装过程交互、文件部署、卸载入口与安装阶段依赖拉取控制。

### 2.2 运行环境原则

- 用户机器不需要预装 Python。
- 用户机器不需要预装 pip。
- 用户机器不需要手动执行 [`requirements.txt`](requirements.txt) 安装。
- 应用运行所需 Python 解释器、Python 依赖与主程序资源均由打包结果自行携带。

### 2.3 大型后端工具分发原则

- [`bin/`](bin) 下大型后端工具不作为完整安装包内置固定内容。
- 大型后端工具采取 安装时下载 或 首次启动时下载 的策略。
- 下载行为必须建立在用户显式确认基础上，不能静默联网拉取。
- 工具下载完成后必须写入既定安装目录，并满足后续主程序直接调用。

### 2.4 依赖确认交互总则

- 安装前必须弹出依赖确认框。
- 仅当用户确认后，安装器才允许下载缺失的大型依赖。
- 若用户取消或拒绝，则安装器直接退出，不继续安装流程。
- 首次启动时若仍存在缺失依赖，也必须弹出同类确认框。
- 首次启动确认框中，若用户拒绝，则程序直接退出，不进入主界面。
- 安装阶段与首次启动阶段的确认逻辑、文案风格、风险提示应保持一致。

## 3. 目标产物与部署方式

## 3.1 安装包目标

交付一套 Windows 桌面安装包方案，满足以下条件：

- 用户双击安装器即可完成安装。
- 安装完成后可通过开始菜单或桌面快捷方式启动。
- 主程序自身可独立运行，不依赖系统 Python 环境。
- 当检测到大型后端工具缺失时，可在用户确认后联网下载并部署。
- 卸载时能够移除程序主体与安装器创建的标准入口。

### 3.2 PyInstaller 产物定位

PyInstaller one-dir 产物作为 Inno Setup 的输入目录，至少包含：

- 主程序可执行文件
- Python 运行时与依赖库
- UI 资源
- 内置基础脚本
- 非大型且必须随包分发的配置模板

### 3.3 安装器定位

Inno Setup 安装器负责：

- 部署 one-dir 程序目录
- 创建开始菜单与桌面快捷方式
- 在安装完成前执行依赖检查与确认
- 在获得授权后下载缺失的大型后端工具
- 记录安装目录与必要元信息
- 提供卸载入口

## 4. 安装目录结构定稿

以下为安装后的标准目录结构：

```text
{app}/
├─ M3U8D.exe
├─ _internal/
├─ config/
│  ├─ config.json
│  └─ manifests/
├─ bin/
│  ├─ aria2c.exe
│  ├─ ffmpeg.exe
│  ├─ N_m3u8DL-RE.exe
│  ├─ yt-dlp.exe
│  ├─ streamlink.exe
│  └─ deno.exe
├─ scripts/
├─ logs/
├─ cookies/
├─ Temp/
└─ uninstall.exe
```

### 4.1 目录结构规则

- `{app}` 表示 Inno Setup 最终安装目录。
- 主程序入口文件位于 `{app}/M3U8D.exe`。
- PyInstaller one-dir 运行时目录保持在 `{app}` 下，不再要求用户可见 Python 环境。
- [`bin/`](bin) 用于存放所有可执行后端工具。
- `config/` 用于持久化程序配置与下载清单元信息。
- `logs/` 用于保存运行日志。
- `cookies/` 用于保存站点 cookie 文件。
- `Temp/` 用于临时解压、下载缓存、工具安装中间文件。
- `scripts/` 用于保留安装后仍需调用的辅助脚本，且仅保留实际需要随安装产物分发的脚本。

### 4.2 对现有仓库目录的映射原则

- 仓库根目录中的运行所需资源在打包后应被归并到安装目录统一结构下。
- 当前开发态 [`bin/`](bin) 中体积较大的可执行文件在安装产物中不默认内置，而是转为按需下载部署到安装目录 [`bin/`](bin) 对应位置。
- 当前开发态 [`config.json`](config.json) 不应继续直接以仓库根路径为运行时唯一路径，而应在安装后映射为 `{app}/config/config.json`。

## 5. 路径规则定稿

### 5.1 路径基准

运行时所有关键路径均以应用安装根目录为基准推导，不允许依赖当前工作目录。

### 5.2 路径规则

- 主程序根路径：`{app}`
- 工具目录：`{app}/bin`
- 配置目录：`{app}/config`
- 主配置文件：`{app}/config/config.json`
- 清单目录：`{app}/config/manifests`
- 日志目录：`{app}/logs`
- Cookie 目录：`{app}/cookies`
- 临时目录：`{app}/Temp`
- 安装后辅助脚本目录：`{app}/scripts`

### 5.3 约束要求

- 代码中不得再写死开发态相对路径作为正式安装版运行依据。
- 所有工具路径应通过统一路径解析函数获取。
- 安装器、启动阶段检查逻辑、主程序工具调用逻辑，必须共享同一套目标路径规则。

## 6. 配置职责定稿

### 6.1 主配置文件职责

[`config/config.json`](config/config.json) 负责保存应用运行配置，但不承担复杂安装状态数据库职责。

建议保留或新增以下职责字段：

- 应用基础设置
- 下载目录与用户偏好
- 浏览器或嗅探相关开关
- 工具目录基准信息
- 是否允许自动检测缺失依赖
- 最近一次依赖检查结果摘要

### 6.2 工具清单职责

在 `{app}/config/manifests` 下新增工具清单文件，用于描述大型后端工具的安装状态与来源信息。其职责包括：

- 工具标识
- 目标文件名
- 目标落盘路径
- 下载地址或发布源标识
- 版本号
- 校验值或校验方式
- 已安装状态
- 最近一次安装时间

### 6.3 职责边界

- 主配置文件偏向用户配置。
- 工具清单偏向安装状态与部署元信息。
- 安装器不应将大量临时流程状态写入主配置文件。
- 启动阶段只读取最小必要状态，不依赖脆弱的临时标记。

## 7. 大型后端工具范围定稿

以下工具纳入大型后端工具管理范围：

- [`bin/aria2c.exe`](bin/aria2c.exe)
- [`bin/ffmpeg.exe`](bin/ffmpeg.exe)
- [`bin/N_m3u8DL-RE.exe`](bin/N_m3u8DL-RE.exe)
- [`bin/yt-dlp.exe`](bin/yt-dlp.exe)
- [`bin/streamlink.exe`](bin/streamlink.exe)
- [`bin/deno.exe`](bin/deno.exe)

### 7.1 分发策略

- 安装器阶段优先尝试完成缺失工具下载。
- 若安装阶段未完成全部工具部署，则首次启动阶段继续补齐。
- 首次启动补齐仍必须经过用户确认。

### 7.2 成功判定

工具下载并部署成功至少满足：

- 目标文件存在于 `{app}/bin`
- 文件大小非零
- 校验通过或版本探测通过
- 对应清单状态已更新为已安装

## 8. 依赖确认框定稿

## 8.1 安装前确认框

安装器在正式复制文件前或进入依赖下载阶段前，必须弹出依赖确认框，至少说明：

- 将要联网下载的依赖名称
- 下载原因
- 预计安装位置为 `{app}/bin`
- 需要网络连接
- 若取消则安装器退出

确认结果规则：

- 用户点击确认：允许进入下载流程
- 用户点击取消或关闭窗口：安装器退出

### 8.2 首次启动确认框

主程序首次启动且检测到仍有缺失依赖时，必须弹出同类确认框，至少说明：

- 当前缺失的依赖名称
- 缺失依赖会导致程序功能不可用
- 确认后将联网下载并安装到 `{app}/bin`
- 拒绝则程序退出

确认结果规则：

- 用户点击确认：进入工具下载流程
- 用户点击取消、关闭或拒绝：主程序直接退出

### 8.3 文案一致性要求

- 安装器与主程序的确认框核心风险提示保持一致。
- 明确写出联网下载行为。
- 明确写出写入目录。
- 明确写出拒绝后的退出后果。

## 9. 关键接口定稿

以下为后续实现所需的关键接口职责，不限定具体语言细节，但要求职责固定。

### 9.1 路径解析接口

- 统一返回安装根目录
- 统一返回 [`bin/`](bin)、`config/`、`logs/`、`Temp/` 等标准目录路径
- 兼容 PyInstaller one-dir 运行时路径解析

### 9.2 依赖清单读取接口

- 读取工具清单定义
- 返回工具元数据、下载源、校验信息、目标路径

### 9.3 缺失依赖检测接口

- 根据清单与目标落盘路径检测缺失工具
- 返回缺失列表、损坏列表、已安装列表

### 9.4 依赖确认接口

- 接收缺失依赖列表
- 统一弹出确认框
- 返回用户是否同意下载

### 9.5 依赖下载与部署接口

- 根据清单执行联网下载
- 完成解压、重命名、落盘、覆盖策略处理
- 更新安装状态清单
- 返回成功项与失败项

### 9.6 启动前依赖保障接口

- 在主程序进入主界面前调用
- 若依赖齐全则继续启动
- 若依赖缺失则弹确认框并按规则下载
- 若用户拒绝或下载失败达到阻断条件，则退出程序

## 10. 安装器职责定稿

Inno Setup 安装器职责限定如下：

- 部署 PyInstaller one-dir 主体文件
- 创建安装目录
- 创建快捷方式与卸载入口
- 在安装流程中检查大型依赖是否需要下载
- 弹出依赖确认框并处理同意或拒绝结果
- 在用户同意后调用下载脚本或下载模块完成依赖安装
- 将工具部署到 `{app}/bin`
- 写入基础安装状态或清单文件

### 10.1 安装器明确不负责

- 不负责要求用户安装 Python
- 不负责要求用户安装 pip
- 不负责要求用户手动执行依赖命令
- 不负责承载复杂业务逻辑
- 不负责在用户拒绝联网下载后继续残缺安装

## 11. 脚本职责定稿

### 11.1 打包脚本职责

打包脚本负责：

- 生成 PyInstaller one-dir 输出
- 收集必要资源
- 排除不应内置的大型后端工具
- 生成供安装器使用的发布目录

### 11.2 依赖下载脚本职责

依赖下载脚本负责：

- 按清单下载大型后端工具
- 支持单工具或批量工具下载
- 执行解压与落盘
- 返回可供安装器或主程序判定的结果码
- 记录下载日志

### 11.3 安装器调用脚本职责

安装器调用脚本负责：

- 作为 Inno Setup 与下载逻辑之间的桥接层
- 接收安装目录、工具列表、日志目录等参数
- 将结果回写为安装器可识别的成功或失败状态

### 11.4 启动期检查脚本或模块职责

启动期检查模块负责：

- 检查 `{app}/bin` 下目标工具是否齐备
- 触发确认框
- 在用户同意后执行下载与部署
- 在用户拒绝或关键依赖未补齐时终止程序启动

## 13. 开发顺序定稿

1. 明确安装版目录结构、路径规则与工具清单格式。
2. 完成路径解析层改造，统一安装版路径基准。
3. 完成依赖清单读取、缺失检测、安装状态记录能力。
4. 完成主程序启动前依赖确认与阻断机制。
5. 完成大型后端工具下载与部署脚本。
6. 完成 Inno Setup 安装器脚本与安装前确认流程接入。
7. 完成 PyInstaller one-dir 打包链路与发布目录整理。
8. 完成安装、首次启动、拒绝下载、失败恢复、卸载等场景验证。
9. 更新安装发布文档并固化交付流程。

## 14. 验收标准定稿

### 14.1 安装体验验收

- 用户机器无 Python、无 pip、未安装 [`requirements.txt`](requirements.txt) 依赖时，安装器仍可正常安装主程序。
- 安装器在需要联网拉取大型依赖前，必定弹出确认框。
- 用户拒绝确认后，安装器直接退出，不继续安装。

### 14.2 首次启动验收

- 若安装阶段未完成全部大型依赖部署，首次启动时能正确检测缺失项。
- 首次启动必须弹出同类确认框。
- 用户拒绝后，程序直接退出，不进入主界面。
- 用户同意后，程序完成下载部署并可继续启动。

### 14.3 路径与部署验收

- 所有运行时关键路径均指向安装目录，不依赖开发态当前目录。
- 所有大型工具最终部署到 `{app}/bin`。
- 配置与清单写入 `{app}/config`。
- 日志输出到 `{app}/logs`。

### 14.4 健壮性验收

- 已安装工具不会被重复下载。
- 工具损坏或缺失时能够被重新识别并触发修复下载。
- 下载失败时能给出明确结果，并阻止进入依赖不满足的运行状态。
- 卸载后主程序主体与标准安装入口被移除。

### 14.5 交付验收

- 产出 PyInstaller one-dir 发布目录。
- 产出 Inno Setup 安装器。
- 产出与该方案一致的安装发布说明。
- 方案范围、职责边界、文件改动面与验收口径均可被后续开发直接执行。

## 15. 实施边界与冻结结论

### 15.1 本次已冻结结论

- 打包组合固定为 PyInstaller + Inno Setup。
- PyInstaller 形态固定为 one-dir。
- 用户环境不再承担 Python、pip、[`requirements.txt`](requirements.txt) 安装责任。
- [`bin/`](bin) 下大型后端工具改为安装时或首次启动时联网下载。
- 下载前必须明确征得用户同意。
- 用户拒绝时，安装器退出或程序退出，不允许带缺依赖继续运行。

### 15.2 后续实现不得偏离

除非重新修订本定稿文档，否则后续开发、脚本设计、安装器实现与验收均以本文档为准。
